Mode d'Emploi du Développeur pour GenJ

L'OBTENIR

Le code source de GenJ est hébergé sur SourceForge à l'adresse suivante : http://genj.sourceforge.net. En utilisant le logiciel cvs/ssh (qui sont des outils pour controler la conformité et mise à jour d'un code détenu dans différents endroits), vous pouvez en obtenir une copie instantanément. Le principal module de développement est ./dev.
Dés à présent, je suppose que vous avez le logiciel cvs/ssh correctement installé sur votre machine. Etablissez les paramètres d'environnement suivants et entrez en ligne de commande dans un répertoire vide de votre choix et tapez :
>cd d:\genj (sous Linux ce sera plutôt cd /home/fm/CVS/genj par exemple)
>set cvs_rsh=ssh
>set cvsroot=:ext:YOURUSERID@cvs.genj.sourceforge.net:/cvsroot/genj
>cvs checkout dev

Si vous n'avez pas été accrédité par l'auteur du projet comme étant un développeur et donc ne pouvez pas remonter sur Sourceforge du code modifié, vous pouvez toujours vous loguer comme anonyme (anonymous) et ainsi reprendre le code source. Il vous suffit pour cela de mettre en "YOURUSERID" le mot "anonymous" et lorsque l'on vous demandera un mot de passe, de faire simplement 'Entrée' (donc ne rien mettre comme mot de passe).

Ces commandes vont placer toutes les sources du module récupéré (ici "dev") dans un sous répertoire ./dev - toutes les données sont maintenant sur votre machine et vont vous permettre de reconstruire l'application (bien sûr vous pouvez commencer par jeter un coup d'oeil à ce qui a déjà été réalisé).

EN Y REGARDANT DE PLUS PRES

Ce qui a déjà été fait sous ./dev consiste dans les éléments suivants :
    ./build.cmd - Script de lancement de ANT sous OS/2
    ./build.sh - Script de lancement de ANT sous Linux
    ./build.properties - Les propriétés du projet (modifiez le chemin pour dir.out)
    ./build.xml - Le construction ANT du projet
    ./doc - Les Fichiers de Documentation
    ./gedcom - Les fichiers Gedcom
    ./help - Les fichiers d'Aide
    ./images - Les fichiers Images
    ./language - Les fichiers ressource des différents langages supportés
    ./lib - Les Librairies
    ./script - Les Scripts (pour construire la distribution finale)
    ./src - Le Source

INSTALLER TOUT CE QUI EST NÉCESSAIRE

Récupérez et installez le logiciel ANT - vous pouvez l'obtenir à l'URL suivante : http://jakarta.apache.org/ant
Assurez vous de bien éditer le fichier ant.properties et d'adapter les paramètres suivants à votre environnement :
    dir.dist - L'endroit où mettre les fichiers qui font parties de la distribution
    dir.build - L'endroit où mettre les fichiers créés durant la compilation
    dir.run - L'endroit où mettre les fichiers permettant de faire tourner l'application sans l'installer
    [par défaut, elles sont définies relativement à dir.out]
(en principe, seul dir.out devrait être modifié).

Important : Assurez-vous que :
- JAVA_HOME pointe sur votre installation JDK/JRE
- ANT_HOME pointe sur votre installation ANT

TRAVAILLER AVEC

Vous pouvez utiliser build.cmd (ou build.sh sous Linux par ex.) pour effectuer un certain nombre d'opérations qui sont nécessaires pour compiler/utiliser GenJ sur votre propre machine. Pour cela il vous suffit de lancer build.cmd (ou build.sh) (ou mettez le dans votre chemin d'environnement) avec l'un des paramètres suivants :
    init - Pour initialiser l'environnement (cela crée les répertoires nécessaires)
    compile.core - Pour compiler le source de GenJ
    compile.report - Pour compiler les rapports
    compile.awtx - Pour compiler l'ensemble AWTx (librairie pour l'interface graphique spécifique)
    compile - Pour tout compiler
    clean - Pour effacer les fichiers temporaires
    version - Pour établir le numéro de version
    dist - Pour faire l'ensemble des fichiers nécessaires à la distribution du programme (app, applet, en, de, fr, help, ...)
    javadoc - Pour créer la documentation Java à partir du source
    run - Pour lancer GenJ à partir du source sans l'installer
    classpath - Pour connaître le classpath
    about - Pour afficher la décharge de responsabilité
[en lançant 'build.cmd run' (ou ./build run (sous linux)) - cela fera tout ce qui est nécessaire pour vous permettre de faire tourner GenJ]

SI VOUS UTILISEZ UN IDE (Interface Graphique de Développement)

Si vous essayez de faire tourner GenJ par l'intermédiaire de votre IDE favori (ex. JBuilder ou Eclipse), il se peut que vous ne souhaitiez pas lancer ant pour créer un ensemble fonctionnel à chaque fois que vous recompilez et relancez.
Vous pouvez ajouter les paramètres suivants à la ligne de commande pour corriger cette situation :
    -Duser.home=[home_folder] indique où sont les fichiers .properties
    -Dgenj.report.dir=[report_folder] indique à partir d'où les rapports doivent être chargés
    -Dgenj.help.dir=[help_folder] indique où le logiciel peut trouver l'aide
    -Dgenj.gedcom.dir=[gedcom_folder] indique où le programme va pouvoir trouver les fichiers gedcom
Voici un exemple:
java -Duser.home=d:\nils -Dgenj.gedcom.dir=d:\nils\gedcoms genj.App
